How to fix a wrong Character/Chapter Association

When the chapter-association scanner picks the wrong character for a name (e.g. "Liu Bei" assigned to a Liu Bei who's the wrong historical period), you can swap it out without losing the chapter ↔ character link.

  1. Open Admin → Character/Chapter Association.
  2. Pick the chapter from the dropdown.
  3. Find the wrongly-attached character in the table.
  4. Click Switch character in that row's Actions column. A small inline form appears.
  5. Type the correct character name. The autocomplete annotates each option with [Faction] [Role] so you can tell duplicate-named characters apart. Each option's value carries a trailing #<id> for disambiguation.
  6. Click Apply.

Behind the scenes: the chapter ↔ old character M2M row is removed and a new chapter ↔ new character row is inserted in the same transaction. The chapter's text and any aliases stay untouched — if the wrong character had a unique alias matched in that chapter, you may also want to clean up that alias on the old character's edit page.

How to add a Character/Chapter Association

Use this when the chapter prose mentions a character by a name we don't know about yet (e.g. a courtesy name, a nickname), or when you want to change which words trigger the pill in this specific chapter. Keywords are stored per-(chapter, character) — editing them here doesn't affect any other chapter.

  1. Open Admin → Character/Chapter Association.
  2. Pick the chapter.
  3. Scroll to Add Character Association to chapter N at the bottom of the page.
  4. In Keywords, enter one or more comma-separated names exactly as they appear in the prose (e.g. Mengde, Cao Cao).
  5. In Canonical character, pick the real character from the dropdown (autocomplete: type partial name, the list shows faction + role + dates). Picking from the dropdown auto-fills the Keywords field with name, courtesy_name, aliases… if you've left it blank — saves retyping.
  6. Click Add.

For each keyword the server scans the chapter text; matching keywords are stored on the chapter_character.keywords row (per-(chapter, character) — see the Keywords column in the table above). Keywords that don't appear are reported and skipped. The chapter ↔ character link is added if it wasn't already there.

Resync semantics. Submitting the form for a character who's already linked to this chapter replaces that pair's keyword list with what you typed — so you can tighten (or drop) which words tag as this character in this chapter without affecting other chapters. The character's global aliases column is never touched by this form. Per-snippet exclusions (× on individual matches) survive resyncs.

Empty submission is allowed — the character stays linked but no inline pills render in the prose (they'll still appear in the chapter sidebar list).

How to add a character

  1. Open Characters.
  2. Click the + Add Character button in the top right (admin-only, hidden otherwise).
  3. Fill in Name (the only required field). Everything else can be filled in later.
  4. If you set a Primary faction that isn't already in the Factions multi-select, the server auto-adds it on save so the data stays consistent.
  5. Click Submit. You'll be redirected to the new character's edit page where you can attach images, URLs, etc.

Duplicate guard: the database has a unique constraint on (name, birth_date, death_date, ancestral_home). If you try to create a second character with all four identical to an existing one, you'll get a friendly flash message instead of a 500.

{# ------------------------------------------------------------------- #}

How to edit a character

  1. From Characters, click the character's name in the table — when you're logged in as admin the name is a link to the edit page. Shortcut: in the chapter sidebar's Character Info panel there's a small Edit button next to each character's name (admin only) that jumps straight to the same page.
  2. Edit any field. Press Submit.

On the edit page you can also:

  • Upload a new image with a source credit — see How to upload a portrait below.
  • Set a portrait as default (auto-makes it visible).
  • Hide / unhide an image.
  • Attach external links — see Linking URLs to a Character.

Changing the character's name, courtesty_name, or aliases triggers an automatic recount of book_mention_count in the same transaction, and the recount also fires on every add / remove / switch on the Character/Chapter Association page and on any rescrape — the chapter sidebar's "N mentions in the book" number stays accurate without a manual flask recount-book-mentions run.

How to upload a portrait (with source credit)

Manual image upload lives in the Images fieldset on the character edit page. Max 10 MB. JPG / JPEG / PNG / GIF / WEBP only — the server sniffs magic bytes, not the extension.

  1. Open the character's edit page.
  2. Scroll to Images → Upload a new image.
  3. Pick the file.
  4. Source site — the credit label shown next to the image (e.g. koei.fandom.com, Wikipedia, Personal photo). Leave blank to fall back to Manual upload.
  5. Source URL — optional link back to the original. If set, the chapter sidebar's Source line renders the site label as a clickable link to this URL; otherwise the label is plain text.
  6. Optionally type a tag name (existing or new — the tag is auto-created), tick Set as default, and pick visibility.
  7. Click Upload.

How to add an event

  1. Open Events.
  2. Click + Add Event.
  3. Fill in Name (required). Aliases / keywords is a comma-delimited list of search terms the Event/Chapter Association tool scans chapters for.
  4. Optionally pick a Location, an Event type (coloured/iconed category — see Event Types), and a Date (free-form text, accepts BC years and ranges).
  5. Geo-point override is a free-form string used when the linked location is wrong/missing for this specific event.
  6. Tick Hide on map to keep the event off the future map render (it'll still appear in the events list).
  7. Submit.
{# ------------------------------------------------------------------- #}

How to edit an event

  1. On Events, click the event name.
  2. Edit, submit.
  3. The edit page has an External links section for attaching URLs.

How to add a faction

  1. Open Factions.
  2. Click + Add Faction in the top right (admin-only).
  3. Fill in Name (required, unique). Optionally tweak the three colours and an icon string.
  4. Submit. You're redirected to the new faction's edit page.

Bulk colour randomization is available via flask randomize-faction-colours — useful after creating a batch of new factions.

There's no FactionAssociation concept (no chapter ↔ faction direct link). Factions show up in a chapter via the characters that appear there.

{# ------------------------------------------------------------------- #}

How to edit a faction

  1. Open Factions.
  2. Each row has an Edit button on the right (admin-only).
  3. Change the name, colours, or icon. Submit.

Faction name is the link to the characters listing filtered by that faction — convenient for double-checking the roster after edits.

How to merge a faction

Use this when two factions are actually the same (different scrape passes, different spelling). The source disappears from all listings; the target absorbs everything.

  1. Open the source faction's edit page (Factions → Edit).
  2. Scroll to the Merge into another faction fieldset at the bottom.
  3. Type the target faction. Autocomplete shows every other non-hidden faction.
  4. Click Merge & hide. Confirm the dialog.

What the merge does atomically:

  • For every character whose M2M factions list contains the source, the target is added too (membership carried over). The source stays in the M2M for history, but won't render anywhere because it's hidden.
  • For every character whose primary faction was the source, the primary is switched to the target.
  • The source faction's is_hidden flag is set true, removing it from all faction dropdowns + listings.

Merge is irreversible from the UI. If you want to undo, set UPDATE faction SET is_hidden = false WHERE id = ... via psql — the M2M membership rows are still there.

How to change a character's default picture

The default picture is the one that loads first in the chapter sidebar's image tabs and shows next to the character row on the characters list.

Two ways, same effect:

From the character edit page
  1. Open the character's edit page.
  2. In the Images grid, find the picture you want.
  3. Click Set default on its card.
From the Image Manager
  1. Open Admin → Image Manager.
  2. Filter to that character (the search box uses the type-ahead picker).
  3. Find the image, click Set default.

Setting default also force-unhides the image (defaults are always public). The previously-default image keeps its visibility — it just loses the default badge. A partial unique index in Postgres enforces "at most one default per character" so this can't drift.

How to hide / remove an image for a character

Hiding makes the image stop appearing on the public-facing pages (chapter sidebar, character edit page outside the Images grid), but the row stays in the DB and the file stays on disk. Same as deletion but reversible from the UI.

To hide
  1. Open the character's edit page or the Image Manager.
  2. Click Hide on the image. The label flips to Unhide for the next click.

If hiding an image that was the default, the default flag is cleared at the same time (you can't have a hidden default).

To permanently remove

There's no Delete button in the UI right now — we intentionally took it out to prevent accidental clicks. If you really need to delete an image: 

-- as the postgres superuser, from stateful_boilerplate:
docker compose exec -T postgres psql -U rotk_app -d rotk_net \
  -c "DELETE FROM portrait WHERE id = 123;"

-- and (optionally) remove the file from disk:
docker compose exec app rm app/static/portraits/<filename>

If you find yourself doing this often, the delete button could come back behind a confirmation dialog — ask.

Event/Chapter + Location/Chapter Association tools

Three association editors today, all the same shape (linked from the admin dropdown):

  • Character/Chapter Association — see fix / add. Per-snippet exclusions also live on this page — see Excluding bad character snippets. When two characters share a name in the same chapter, see Duplicate-name characters.
  • Event/Chapter Association — pick the event from the dropdown; the picker auto-fills the Keywords field with the event's name+aliases. Keywords are optional — leave blank to attach the event to the chapter without scanning the prose (event still shows in the chapter sidebar, just no inline tagging).
  • Location/Chapter Association — same picker + keyword pattern as Event/Chapter. Locations pinned via an Event's location FK also show up in the chapter sidebar automatically — this tool is for places that get referenced independent of any specific event. Per-snippet exclusions live on this page — see Excluding bad location snippets.

Faction ↔ chapter is intentionally NOT a thing — factions are referenced via the characters that appear in the chapter (so changing a character's primary faction is enough to get them into the right faction's listings).

Adding URL types

URL types are categories for external links (e.g. Wikipedia, YouTube, Wikidata). Each type carries a Font Awesome icon class that renders next to matching URLs in lists and the chapter sidebar.

  1. Open Admin → URL Types.
  2. Click New URL type.
  3. Set Name (required, unique).
  4. Set Font Awesome icon class. Browse fontawesome.com and copy the class names from any free icon, e.g. fa-brands fa-wikipedia-w or fa-solid fa-link.
  5. Optional: pick colours (not currently rendered in the UI; reserved for badge styling later).
  6. Submit. On the edit page you'll see a live icon preview.

Deletion is blocked while any URL still references the type — either reassign those URLs first, or wait until they're gone.

Excluding bad location snippets in a chapter

When a location's name or alias accidentally matches part of another word in the prose (e.g. an alias "Liang" matching a character named "Zhang Liang"), you don't have to remove the alias entirely — you can suppress just the wrong occurrence.

  1. Go to Location/Chapter Association and pick the chapter.
  2. For each associated location, click show N snippets to expand the live match list.
  3. Hit the red × next to a snippet to mark it as a bad match. A confirm dialog appears first.
  4. The snippet moves into an Excluded snippets collapsible below — strikethrough, with a restore (↺) button if you change your mind. No page reload: both pools update live as you click.

The exclusion is fingerprinted by (chapter, location, the surrounding text). Once excluded, that specific occurrence stops appearing as an inline ref in the chapter prose too — the rest of the location's matches continue to be tagged normally.

Where: Admin → Location/Chapter Association. Pick a chapter, expand the snippet list on the row, click the red × next to any bad match.

Event Types (creating + assigning)

An Event Type is a coloured/iconed category for events (e.g. Battle, Treaty, Court intrigue). It appears as a badge next to the event everywhere the event is shown — chapter sidebar, events list, etc. Same shape as URL Types: name + Font Awesome icon + three colours.

Create a new event type

  1. Go to Event Types and click New event type.
  2. Give it a name (required), pick a Font Awesome icon class — browse fontawesome.com and copy the class string (e.g. fa-solid fa-flag).
  3. Optionally set background / font / border colours. The randomize button generates a readable palette in one click.
  4. Save.

Assign an event type to an event

  1. Edit the event (Events → Edit row).
  2. Pick from the Event type dropdown — leaving it blank means no badge.
  3. Save. The badge now appears in the chapter sidebar's Events accordion for every chapter the event is associated with.

Where: Admin → Event Types for managing types; Events → Edit for assigning.

Chapter page reader controls (Tag style + Link style)

Two dropdowns sit next to the chapter title at the top of every chapter page — reader preferences, persisted client-side so they carry across chapter navigation. Useful to know about when triaging UI questions from readers.

Tag style

Controls how inline character refs render in the prose: Rounded pills (default), Square boxes, Underline + coloured, or Coloured only. Persisted in localStorage under rotk.chapterStyle. Event and Location refs are always plain-black-underlined regardless of the choice.

Link style

Controls how clicking (or hovering) a character ref reveals the sidebar panel: Click (default) or Hover. Desktop-only — the dropdown is hidden on mobile/tablet, where tapping always opens a modal regardless. Persisted in a real cookie (rotk_link_style, 1-year expiry).

Inline event/location refs

Black-underlined words in the prose are events / locations associated with the chapter. Clicking opens the matching sidebar accordion (Events / Locations) and briefly flashes the specific row — no page scroll.

Excluding bad character snippets in a chapter

Full parity with the location snippet-exclusion flow — when a character's keyword accidentally matches part of another word, you can suppress just that occurrence without removing the keyword from the whole chapter.

  1. Go to Character/Chapter Association and pick the chapter.
  2. Find the character. Click show N snippets to expand the live match list.
  3. Hit the red × next to a snippet to mark it as a wrong match. A confirm dialog appears first.
  4. The snippet moves into an Excluded snippets collapsible below — strikethrough, with a restore (↺) button. No page reload; both pools update live.

The stored exclusion is a fingerprint of the surrounding text (~60 chars before + the matched needle + ~60 chars after, whitespace-normalised). At render time the chapter view recomputes the same fingerprint per match and skips any that are in the exclusion set. So the pill stops appearing in the chapter prose too.

Where: Admin → Character/Chapter Association. Pick a chapter, expand the snippet list on the character's row, click the red × next to any bad match.

Duplicate-name characters in a chapter (splitting matches)

When two different characters share a name (e.g. two "Lady Cao" rows both associated with chapter 16), the chapter view can't tell from a needle alone which one an occurrence belongs to. The renderer's default is "first-registered character wins" — but you can split the occurrences deliberately by mirroring exclusion sets.

Spotting the duplicate

Any pill in the prose whose name is shared by another character in the same chapter gets a small red badge next to it (admin-only). Clicking that badge jumps straight to the Character/Chapter Association editor for that chapter.

Splitting matches

Say the chapter has 4 "Lady Cao" matches and you want A to own matches 0 and 2, B to own 1 and 3:

  1. On character A's row in the association editor, exclude occurrences 1 and 3 (the ones that should be B).
  2. On character B's row, exclude occurrences 0 and 2 (the ones that should be A).
  3. The chapter render walks candidates per occurrence: for each match, the first character who hasn't excluded that occurrence gets the pill. So occurrence 0 goes to A (didn't exclude), 1 to B (A excluded → walk on), 2 to A, 3 to B.

If both characters exclude the same occurrence, the pill drops — the words render as plain prose. If neither excludes, the first-registered character wins by default.

Chapter dates in the header

Every chapter row has a free-form date column (VARCHAR(64), same shape as character birth/death dates). When set, the chapter view renders it as a small muted line directly under the chapter title — e.g. 192-200 AD. When empty, no line is shown.

Setting the date

There's no admin form for individual chapters — dates are managed in bulk via CLI so an LLM-assisted dating pass can write them out and apply them together:

  • flask dump-chapters-for-dating <start> [end] — dump prose + neighbouring context as JSON, pipe into your dating tool.
  • flask apply-chapter-dates FILE [--apply] — apply [{chapter_num, date}] JSON to chapter.date. Dry-run by default; pass --apply to write.

The column is a display string, not a parsed range — arbitrary formats work ("208 AD", "Winter 208–Spring 209", "BC 195-190"). Timeline / sortability work uses tools.date_parser on top of this string; run flask check-date-parsing to see which strings the parser currently can't handle.

Re-scraping chapters after a source fix

Use this when the scraper's been updated and you need to refresh chapter prose without disturbing any admin work. Never do a delete-and-rescrape — the FKs from chapter_character, event_chapter, chapter_location, and match_exclusion all cascade on chapter deletes and you'd lose every association + per-chapter keyword + excluded snippet for those chapters.

  • flask rescrape-chapter <n> — re-fetch one chapter and UPDATE its name + content in place. Associations, keywords, and MatchExclusion rows are untouched (chapter row keeps the same id). Every character associated with the chapter gets a book_mention_count recount at the end.
  • flask rescrape-all-chapters — loops the same over every chapter. Skips unchanged chapters (prints unchanged). Recounts affected characters once at the end, deduped.

Caveat: per-snippet MatchExclusion rows are fingerprinted by the ~60 chars of text around the match. If a rescrape recovers a paragraph whose text lands within that window, the stored fingerprint won't match the regenerated one, and the exclusion silently stops applying (the row stays in the DB — it's just orphaned). Re-× the affected snippets in the admin UI to re-fingerprint against the new content. Only exclusions whose before or after window actually crosses the insertion are at risk; most survive.

Backfilling per-association keywords (one-shot after migration 0012)

Migration 0012_per_association_keywords.sql added a keywords column to chapter_character, event_chapter, and chapter_location. The column starts empty on every existing row — the chapter renderer falls back to the entity's global aliases in that case, so nothing visibly breaks, but the per-chapter keyword model isn't in effect until the column is seeded.

  1. flask apply-migrations — applies 0012 (adds the empty column).
  2. flask backfill-association-keywords --dry-run — preview: shows how many rows would be seeded and a sample of the generated keyword strings.
  3. flask backfill-association-keywords — seeds every association row whose keywords is still empty. Each character row gets "name,courtesy_name,alias1,alias2,…" (whitespace-stripped, deduped, no leading spaces); events and locations get "name,alias1,alias2,…". Rows already populated via the admin UI are left alone.

After that, every admin submission on the Character/Event/Location Association page writes to the association row's keywords column, not the global aliases. Resync semantics are per-chapter: editing keywords for chapter 5 doesn't affect chapters 1-4 or 6+.

Idempotent — safe to re-run. Only touches rows with an empty keywords column.
